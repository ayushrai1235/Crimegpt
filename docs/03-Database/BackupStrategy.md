# Backup Strategy

## Overview
The **Backup Strategy** document outlines the protocols for preserving the integrity and availability of **CrimeGPT** data. Given that the platform houses sensitive law enforcement intelligence, data loss is unacceptable. 

This strategy leverages native **Zoho Catalyst** backup capabilities while defining off-platform redundancy for external databases.

---

## 1. Backup Tiers

Backups are categorized into three tiers based on the criticality of the data and the required Recovery Point Objective (RPO).

### Tier 1: Highly Critical (Irreplaceable Data)
- **Data:** Audit Logs, Chat Session History, FIR Metadata.
- **Location:** Catalyst Data Store.
- **Strategy:** Automated hourly snapshots.

### Tier 2: Critical (Replaceable but Expensive to Rebuild)
- **Data:** Raw FIR PDFs, Evidence Images.
- **Location:** Catalyst File Store.
- **Strategy:** Daily incremental backups.

### Tier 3: Derived Data (Can be regenerated)
- **Data:** Graph Relationships (Neo4j), Text Embeddings (Vector DB), Heatmaps (Catalyst Cache).
- **Strategy:** Daily snapshots. If lost, this data can technically be regenerated by re-running the Ingestion Pipeline on the raw PDFs, though this is compute-intensive and slow.

## 2. Zoho Catalyst Backup Implementation

### 2.1. Catalyst Data Store
- **Automated Snapshots:** Zoho Catalyst provides automated backups for the relational data store based on the project's subscription tier. We will configure this for the maximum frequency available (e.g., daily or hourly).
- **Custom Cron Backup:** To guarantee our 1-Hour RPO, a **Catalyst Cron** function will be written to execute a `SELECT *` (or delta extraction) of critical tables (like `Audit_Logs`) and save the output as an encrypted JSON file in a secure Catalyst File Store bucket.

### 2.2. Catalyst File Store
- **Redundancy:** Zoho Catalyst inherently replicates object storage across multiple physical drives within the data center.
- **Versioning:** Object versioning must be enabled on the File Store buckets. If a malicious actor overwrites an evidence PDF with a blank file, the previous version can be instantly restored without reverting the entire system.

## 3. External Database Backup Implementation

### 3.1. Neo4j (Graph Database)
- **Managed Backups:** Using Neo4j AuraDB Enterprise ensures automated, continuous backups are running in the background without impacting query performance.
- **Retention:** 30 days of rolling backups.

### 3.2. Vector Database
- **Managed Backups:** Managed providers (Pinecone/Weaviate) provide automated collection snapshots.
- **Retention:** Weekly snapshots. Since the Vector DB only contains mathematical representations of the FIRs (which are safely stored in Catalyst), point-in-time recovery is less critical than for the relational data.

## 4. Off-Site Cold Storage (The "Doomsday" Protocol)

To protect against a catastrophic event that wipes out the primary cloud provider entirely, an air-gapped backup strategy is required.

- **Process:** Weekly, a highly secure **Catalyst Function** triggers. It compiles a full export of the Catalyst Data Store, encrypts it using a KSP-managed public key, and transmits it via SFTP to an isolated, on-premise server located within KSP Headquarters.
- **Security:** This file can only be decrypted using a private key held on a physical smart card controlled by the KSP IT Director.
