# Business Rules

## Overview
The **Business Rules** document defines the strict, non-negotiable logic constraints that govern how the **CrimeGPT** system operates. These rules ensure legal compliance, data integrity, and operational safety. All business rules MUST be enforced on the backend via **Catalyst Functions** and database constraints.

---

## 1. Access and Visibility Rules

### BR-1.1: Data Jurisdiction
- **Rule:** A Station House Officer (SHO) can only view detailed FIR evidence and case notes for crimes that occurred within their assigned geographical jurisdiction (Police Station limits).
- **Exception:** If an SHO is explicitly added as a "Collaborating Officer" to a case outside their jurisdiction by a DSP or higher.
- **Enforcement:** Catalyst Functions must filter all database queries by the user's `station_id` unless their role is `Commissioner` or `Analyst`.

### BR-1.2: Audit Immutability
- **Rule:** Once an audit log entry is written, it can NEVER be altered or deleted, even by a System Administrator.
- **Enforcement:** The `audit_logs` table in the **Catalyst Data Store** must be restricted at the database level to only allow `INSERT` and `SELECT` operations. No `UPDATE` or `DELETE` endpoints will be exposed in Catalyst Functions.

## 2. AI and Data Processing Rules

### BR-2.1: Zero Hallucination Mandate
- **Rule:** The Conversational AI (CrimeGPT) is strictly prohibited from generating factual claims about suspects, locations, or dates that are not explicitly present in the provided RAG context.
- **Enforcement:** The LLM prompt engineered in the Catalyst Function must include strict negative constraints (e.g., "If the answer is not in the context, reply 'Information not found'"). The output parser must validate that a citation link is present for every factual sentence.

### BR-2.2: Data Retention
- **Rule:** Chat session histories are retained for active investigations but must be archived after case closure.
- **Enforcement:** A **Catalyst Cron** job running monthly will scan for chat sessions linked to `CLOSED` cases and move them from the primary Catalyst Data Store to cold storage (Catalyst File Store PDFs) to save database space.

## 3. Reporting and Export Rules

### BR-3.1: Watermarking
- **Rule:** Any PDF generated by the system (Intelligence Briefs, Chat Exports) MUST contain a digital watermark detailing the downloading officer's ID, IP address, and the timestamp.
- **Enforcement:** The PDF generation library utilized within the **Catalyst Function** must embed this watermark before saving the file to the **Catalyst File Store** for download.

### BR-3.2: Export Limitations
- **Rule:** Users cannot bulk-download raw FIR datasets. Export is limited to processed reports and individual case files.
- **Enforcement:** Catalyst Functions will enforce rate limiting on the `/api/export` endpoint. Any attempt to export more than 50 files in an hour will trigger a security alert to the Administrator.
